# Checklist — Охота на баги

Интерактивный чеклист для студентов: тестируют API, отмечают найденные баги, проверяют себя.

**Страница:** `public/checklist.html`
**Деплой:** [eddytester.com/checklist.html](http://eddytester.com/checklist.html)

## Как работает

1. Студент вводит API-ключ на экране авторизации
2. Фронтенд проверяет ключ через `/check-key?key=xxx` (прокси на том же сервере)
3. Если ключ валидный — открывается чеклист с 5 эндпоинтами
4. Студент тестирует каждый метод и отмечает найденные баги галочками
5. Нажимает «Проверить»:
   - если есть лишние (фейковые) баги — ❌ ошибка
   - если отмечены не все верные — 🤔 анимация
   - если всё верно — 🎉 фейерверк

## Авторизация

- Экран ввода ключа (без логина/пароля)
- Ключ сохраняется в `localStorage`
- При повторном визите ключ проверяется заново
- Ключ **не отображается** на странице

## Прокси-валидация ключей (`key-check.js`)

`public/key-check.js` — лёгкий Node.js сервер без зависимостей.

```
GET /check-key?key=xxx  →  {"valid": true/false}
```

Прокси дёргает `localhost:3000/v1/api/users` с заголовком `X-Fix-Bug` и проверяет HTTP-статус (200 = валидный).

**Запуск через PM2:** `pm2 start key-check.js --name key-check`

## Apache-прокси

В `/etc/apache2/sites-available/eddytester.com.conf`:

```
ProxyPass /check-key http://localhost:3456/check-key
ProxyPassReverse /check-key http://localhost:3456/check-key
```

Включены модули: `proxy`, `proxy_http`.

## Данные в БД

Для работы нужны:

- `api_keys` — хотя бы один ключ (например, `test-key-123`)
- `users` — тестовые пользователи для GET-запросов

После создания ключа и пользователей перезапуск не требуется — API проверяет через `SELECT`.

## Разработка

Локально ключи проверяются через `SKIP_DB=true` в `.env`. На проде ключи в PostgreSQL.
